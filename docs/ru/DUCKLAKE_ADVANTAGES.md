# Преимущества DuckLake перед другими Lakehouse (Iceberg + Trino)

## 1. SQL-каталог вместо файлов метаданных
- Все метаданные (схемы, снапшоты, статистика) хранятся в реляционной БД (PostgreSQL, SQLite, MySQL).
- Нет «ада метаданных» — тысяч мелких файлов в S3.
- Быстрое планирование запросов (SQL вместо множества HTTP-вызовов к объектному хранилищу).

## 2. Многотабличные ACID-транзакции
- Атомарное обновление нескольких таблиц в одной транзакции (`BEGIN` … `COMMIT`).
- Критично для финансовых, инвентарных и других бизнес-сценариев с целостностью данных.
- Iceberg + Trino поддерживают транзакции только в пределах одной таблицы.

## 3. Простота операций и обслуживания
- Очистка старых снапшотов — обычный `DELETE` в БД, а не сложные `expire snapshots`/`rewrite manifests`.
- Компакция файлов проще благодаря централизованному каталогу.

## 4. Гибкость (Bring Your Own)
- **Любая SQL БД** для каталога (PostgreSQL, SQLite, MySQL, DuckDB).
- **Любое хранилище** для Parquet-файлов (S3, GCS, MinIO, локальная ФС).
- **Любой клиент** (DuckDB, pg_ducklake, DataFusion и др.) — формат не привязан к движку.

## 5. Единый опыт разработки и production
- Локальная разработка на DuckDB/SQLite, production на PostgreSQL + S3 без изменения кода.
- Iceberg + Trino требуют эмуляции S3 и каталога для локальной разработки.

## 6. Низкая задержка при работе с метаданными
- Каталог в БД (часто рядом с клиентом) даёт быстрые SQL-запросы.
- Iceberg вынужден читать множество файлов из S3 даже при кэшировании.

## 7. Не зависит от DuckDB
- DuckLake — это спецификация, а DuckDB — один из возможных клиентов/каталогов.
- Можно использовать PostgreSQL + DataFusion или pg_ducklake без DuckDB.

---

## Компромиссы (кратко)
- **Высокая зависимость от каталога** — при потере БД восстановить таблицы сложнее, чем в Iceberg.
- **Центральное узкое место** — при экстремальной частоте коммитов БД может стать бутылочным горлышком.
- **Меньшая зрелость** экосистемы по сравнению с Iceberg.

---

## Для каких задач DuckLake подходит лучше всего?
- Несколько таблиц, которые должны обновляться согласованно.
- Сценарии, где важна простота эксплуатации метаданных.
- Dev-процессы, требующие идентичного окружения на всех этапах.
- Уже есть PostgreSQL или хочется избежать отдельного сервиса каталога.
